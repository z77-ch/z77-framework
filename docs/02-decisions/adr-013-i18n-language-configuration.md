# ADR-013 ŌĆö i18n Language Configuration

**Status:** `APPROVED`
**Date:** 2026-06-05

---

## Context

z77 carries a `language` dimension end to end ŌĆö `Request` extracts it from the URL
prefix, `MetaData` and `Content` are keyed by `(ŌĆ”, language)`, and `PageIdentity`
caches per language. But the **source of truth for which languages exist and which
one is the default was scattered and implicit**:

- `Request::DEFAULT_LANGUAGE = 'de'` (core routing fallback)
- `MetaDataController::DEFAULT_LANGUAGE = 'de'` (backend CRUD default)
- `Request::extractLanguage()` accepted **any** two-letter alpha segment as a valid
  language ŌĆö there was no registry to validate against, so `/xx/home` was a valid
  route for a non-existent language.

Two needs were on the table:

1. A multi-language picker (backend + frontend) ŌĆö deferred pending in
   `metadata.md` ŌĆö needs to know *which* languages a given environment offers.
2. A single, predictable place to set the system default, because configuration is
   increasingly changed via AI assistants ("set everything to default `en`"). An AI
   (or a successor developer) must find the one hebel immediately and change it
   without hunting through scattered constants and risking an inconsistent state.

The scope assumption: defined languages are **valid for all modules** by default.
But a language switch is not always wanted ŌĆö e.g. an internal `financial` module
has no use for it. So a module must be able to opt out / narrow the set.

## Decision

Introduce a dedicated, semantically named **i18n configuration** as the single
source of truth, with an optional per-module override.

### Global i18n config (source of truth)

A dedicated config namespace `i18n` (own config file, generated by the installer
from the project's installer-config section ŌĆö **not** hand-edited in the generated
file, and **not** folded into `moduleManager`). Two keys:

```php
return [
    'defaultLanguage' => 'de',              // the one hebel: system-wide fallback
    'languages'       => ['de', 'fr', 'it'], // globally available set (the whitelist)
];
```

- `defaultLanguage` is the **single** system-wide fallback. It replaces
  `Request::DEFAULT_LANGUAGE` and `MetaDataController::DEFAULT_LANGUAGE`. "Set
  everything to `en`" = change this one line.
- `languages` is the globally available set and the **validation whitelist** for
  `Request::extractLanguage()` ŌĆö an unknown two-letter prefix is no longer silently
  accepted.

### Two distinct roles for `languages`

The global `config/i18n.languages` is the **system-wide routing whitelist** ŌĆö every
language the system can serve and validate in a URL segment. The per-module
`languages` key (in `App/Config/{module}Config.inc.php`, read through
`ModuleManager::getModuleLanguages()`) is the **switch offering** for that
environment ŌĆö opt-in, a subset of the global whitelist:

| Module config | Meaning |
|---|---|
| `languages` omitted | no switch ŌĆö the environment stays single-language (default) |
| `languages => ['de', 'fr']` | switch offers exactly these (subset of the global whitelist; > 1 entry ŌćÆ switch shown) |

Opt-in (not inherit-global): adding a global language must NOT silently put a switch
on every environment. The admin backend declares no `languages` and therefore stays
single-language with no extra opt-out flag. Each environment states its languages
explicitly (discoverability Ōē½ magic inheritance). The `defaultLanguage` is always
servable everywhere as the fallback, independent of this list.

### Language resolution & persistence

> **Superseded in part by ADR-015 (2026-06-08, Variante B).** The clause below ŌĆö
> "a request without a prefix reads the session" ŌĆö is no longer the rendering rule.
> A prefix-less resolved URL now ALWAYS renders `defaultLanguage`; the session only
> redirects the bare site root (`/` ŌåÆ `/{lang}`), never the rendered language of a
> resolved page. This was required to make the canonical alias path (`/home`)
> language-stable. See ADR-015 ┬¦"Language stability addendum" and
> `docs/topics/i18n.md` I18N-LANG-STABLE-001. The rest of this ADR (the i18n config
> source of truth, whitelist validation, per-module opt-in) stands unchanged.

A valid two-letter URL prefix **sets** the language and **writes it to the session**;
a request without a prefix **reads** the session, falling back to `defaultLanguage`.
The session deliberately starts after routing, so the reconciliation runs in
`Dispatcher::execute()` (after `AccessGuard` started the session) and before the
page-cache decision keys on the language. The switch button is therefore just a link
carrying the target prefix.

### Content language fallback

A content document that does not exist in the requested language falls back to the
`defaultLanguage` document (`ContentService::find()`). An existing-but-inactive
document is a deliberate state and does NOT fall back.

### Why a dedicated `i18n` config and not a scalar inside `moduleManager`

This decision deliberately establishes a reusable principle (see Reasoning):
**every configuration value lives at a semantically discoverable single source.** An
AI or developer asked to change "the language" searches for `language` / `locale` /
`i18n` ŌĆö not for `moduleManager`. Folding the language registry into the module
manager would hide it behind an unrelated name and dilute that service's
responsibility (module / view-area / cache-policy resolution). The cross-cutting
principle is recorded in `conventions.md`; a later audit (roadmap) checks whether it
is carried through the rest of the framework.

## Reasoning

**Why global = source of truth, module = override (not fully per-module)?**
The "set everything to `en`" operation is inherently global. With per-module lists
as the only source, that operation touches N module files and an AI/developer can
miss one ŌåÆ inconsistent state. A single global hebel makes the common global change
atomic and the per-module file the rare, explicit exception.

**Why validate against a whitelist now?**
`extractLanguage()` accepting any two-letter code is a latent correctness hole
(`/xx/home` routes to a non-existent language and falls back silently). The whitelist
that the picker needs anyway closes that hole immediately ŌĆö not on-spec.

**Why is this an ADR and not just a topic note?**
It pins a structural decision (own namespace vs. fold-in; global-with-override vs.
fully distributed) with rejected alternatives, and it is the first concrete
application of the discoverable-single-source principle.

## Consequences

**Easier:**
- One line changes the system default language ŌĆö AI- and human-friendly.
- Unknown language segments are rejected at routing instead of silently defaulting.
- The deferred multi-language picker has a defined data source (global set Ōł®
  per-module override).

**Harder / to keep in mind:**
- A new public view-area module that should be multi-language relies on the global
  `languages`; if it must deviate, it sets `languages` in its module config and that
  subset MUST include `defaultLanguage` (validated).
- The installer gains one more generated config file (`config/i18n.inc.php`); its
  values come from the installer-config source, never hand-edits in the generated
  file (regenerated on reinstall).

**Built on top of this foundation (2026-06-05):**
- Frontend language switch (de/fr): per-module `languages` opt-in, session
  persistence, content language fallback, switch button in the topbar. See
  [`../topics/i18n.md`](../topics/i18n.md).

**Known trade-off:**
- Session-remembered language means every visitor gets a session and the page cache
  keys per language (`PageIdentity` already includes `language`, so no collision) ŌĆö
  the cost of the chosen "session remembers" model. Acceptable for v1.

## Rejected Alternatives

| Option | Why rejected |
|---|---|
| `defaultLanguage` scalar inside `moduleManager` config | Hidden behind an unrelated name; an AI/dev searching "language" won't find it; dilutes ModuleManager's responsibility. Violates the discoverable-single-source principle. |
| Languages defined fully per-module, no global source | "Set everything to X" becomes an N-file edit prone to missing a module ŌåÆ inconsistent state. No single hebel. |
| Per-module `languages` omitted ŌćÆ inherit the global set | Adding a global language would silently put a switch on every environment incl. the admin backend; needs an extra opt-out flag. Opt-in keeps each environment's languages explicit (discoverability). |
| Keep scattered `DEFAULT_LANGUAGE` constants | Multiple sources of the same truth; the exact problem this ADR removes. |
| Hand-edit languages into the generated `moduleManager.inc.php` | File is installer-generated and ephemeral; edits are lost on reinstall. |
