## Context

`FileFinder` resolves file paths across namespace-based lookup trees. It is the backbone of
the override system: every file lookup (source, template, asset) goes through it and checks
the override paths first, then the vendor paths. Registered once at bootstrap, used by
`Bootstrap`, `ConfigManager`, `StylesheetManager`, `LayoutManager`, and `TemplateRenderer`.

The config (`config/fileFinder.inc.php`) is auto-generated by the Installer and maps PHP
namespaces to path arrays. `FileFinder` resolves those paths at runtime, filtering out
non-existent directories, and caches the result for the request lifetime.

## What Works Well

- **Override system concept** — `sourcePaths: [override/z77/core, vendor/z77/kernel/core]` gives
  project-level overrides priority without monkey-patching. Clean and predictable.
- **Three-tier caching** — local property → DataCache → filesystem. First request per process
  parses the config file; subsequent requests hit memory. Correct and fast.
- **Lenient path discovery** — non-existent directories are silently skipped via `realpath()`
  check, so a module without assets doesn't break the bootstrap.
- **`getAbsPath` + `mkdir`** — centralises the `ABS_BASE_PATH`-relative path construction
  with optional directory creation. Used cleanly in Bootstrap for the log directory.
- **`realpath()` for path validation** — correct choice: follows symlinks, normalises
  separators, returns `false` on missing paths. No `file_exists()` + `is_dir()` double check.

## Issues

### BUG-FF001 — `getFirstAssetMatch`: unused `$cfg` variable

**Severity:** Medium — dead code that misleads readers about the method's intent
**File:** [FileFinder.php:113](../../packages/kernel/core/src/Libraries/FileFinder.php#L113)

```php
// Current
public function getFirstAssetMatch(...): ?string {
    $cfg = $this->discoverNamespacePaths();   // ← $cfg is never used
    return $this->findFirstMatch(
        fileName: $fileName,
        nameSpace: $nameSpace,
        resourceType: 'assetPaths',
        ...
    );
}
```

`$cfg` is assigned but never referenced. `findFirstMatch` triggers `discoverNamespacePaths()`
internally via `getBasePaths()`, so the explicit call here is redundant. Unlike
`getFirstSourceMatch` and `getFirstTplMatch` which read `$cfg->getResourceDir()['sourceDir']`
/ `['tplDir']` to build the `subDir` argument, `getFirstAssetMatch` passes no `subDir`.

**Fix:** Remove the `$cfg` assignment.

```php
public function getFirstAssetMatch(...): ?string {
    return $this->findFirstMatch(
        fileName: $fileName,
        nameSpace: $nameSpace,
        resourceType: 'assetPaths',
        throwError: $throwError,
        cachePersist: $cachePersist
    );
}
```

### BUG-FF002 — `getFirstAssetMatch`: missing `bool` type on `$cachePersist`

**Severity:** Medium — type safety gap, inconsistent with all other signatures
**File:** [FileFinder.php:110](../../packages/kernel/core/src/Libraries/FileFinder.php#L110)

```php
// Current
public function getFirstAssetMatch(
    string $fileName,
    string $nameSpace,
    bool $throwError = true,
    $cachePersist = true    // ← missing bool
): ?string {

// Fix
    bool $cachePersist = true
```

Every other method (`getFirstSourceMatch`, `getFirstTplMatch`, `findFirstMatch`) declares
`bool $cachePersist`. This one doesn't. PHP accepts any value silently, and DataCache trusts
the caller — a non-bool would propagate without error.

### BUG-FF003 — Cache key in `findFirstMatch` missing `$resourceType`

**Severity:** Medium — structural correctness; collision possible if API is extended
**File:** [FileFinder.php:81](../../packages/kernel/core/src/Libraries/FileFinder.php#L81)

```php
// Current
$key = [$nameSpace, $subDir, $fileName];

// Fix
$key = [$nameSpace, $resourceType, $subDir, $fileName];
```

The cache key doesn't include `$resourceType`. Two lookups with the same namespace, subDir,
and filename but different resource types (e.g. `sourcePaths` vs. `assetPaths`) would share
a cache slot and the second lookup would return the first result.

In the current implementation this can't happen (source and asset subDirs are always
distinct), but it's a structural flaw: the key doesn't fully describe the lookup. Adding
`$resourceType` makes the key self-documenting and future-proof.

### BUG-FF004 — `assetDir` config key is silently unused

**Severity:** Medium — config lie; confuses successors
**File:** [FileFinder.php:107-120](../../packages/kernel/core/src/Libraries/FileFinder.php#L107-L120),
[skeleton/config/fileFinder.inc.php:9](../../skeleton/config/fileFinder.inc.php#L9)

The config defines:
```php
'resourceDir' => [
    'sourceDir' => 'src',
    'assetDir'  => 'assets',    // ← written by Installer, never read by FileFinder
    'tplDir'    => 'res/view/templates',
],
```

`sourceDir` and `tplDir` are passed as `$subDir` in `getFirstSourceMatch` /
`getFirstTplMatch`. `assetDir` is never read — `getFirstAssetMatch` passes no `subDir`.
Assets are therefore found in the root of the `assetPaths` entries
(`public/assets/frontend/`), not in a subdirectory named `assets/`.

This works correctly today because the `assetPaths` in the config already point to the
final directory (`$baseDir.'public/assets/frontend'`). But `assetDir = 'assets'` sits in the
config doing nothing, and any successor will assume it has the same role as `sourceDir`.

**Decision required:** Either
- (a) remove `assetDir` from the config and Installer — assets don't need a subDir, and the
  current structure is correct as-is, or
- (b) use it as subDir (breaking change for existing asset paths, requires Installer update).

Recommended: **(a)** — remove `assetDir`. Document that asset paths point directly to the
final directory (no subDir), unlike source/template paths.

### QUAL-FF002 — `discoverNamespacePaths()`: misleading name and mixed role

**Severity:** Minor — clarity, open-source readability
**File:** [FileFinder.php:21-66](../../packages/kernel/core/src/Libraries/FileFinder.php#L21-L66)

The method does two things:
1. Parses `fileFinder.inc.php` into a `Config` object.
2. Filters paths by existence and stores the result back into the same `Config` object via
   `$cfg->setNamespacePaths($namespacePaths)`.

The name "discover" implies an active filesystem scan, not a config load. More importantly,
the result (filtered `namespacePaths`) is stored back into the same `Config` object that
holds the raw config data. A single `Config` instance ends up carrying both static config
(`resourceDir`, `namespaces`) and computed state (`namespacePaths`).

**Fix (name):** Rename to `resolveConfig()` — it loads and resolves, both.

**Design note:** The Config-as-state-container pattern is worth refactoring in v1.1 into
two separate objects: an immutable `FileFinderConfig` (parsed from file) and a separate
`ResolvedPaths` value object (computed at boot). For v1.0 the rename is sufficient.

### QUAL-FF005 — Error message conflates "namespace not registered" with "file not found"

**Severity:** Minor — diagnostic quality
**File:** [FileFinder.php:100-103](../../packages/kernel/core/src/Libraries/FileFinder.php#L100-L103)

```php
// Current
$list = $paths ? implode(', ', $paths) : "no {$subDir} paths registered";
throw new \RuntimeException("File '{$fileName}' not found in [{$list}] for namespace '{$nameSpace}'.");
```

When `$paths` is empty (namespace not registered or no paths for this resource type), the
error embeds the explanation inside the "file not found" message. The diagnosis is the
opposite: the file wasn't searched at all because the namespace is unknown to FileFinder.

```php
// Fix
if (empty($paths)) {
    throw new \RuntimeException(
        "Namespace '{$nameSpace}' has no registered {$resourceType} paths. " .
        "Check config/fileFinder.inc.php."
    );
}
throw new \RuntimeException(
    "File '{$fileName}' not found in: " . implode(', ', $paths) . " (namespace '{$nameSpace}')."
);
```

### QUAL-FF006 — Mutable `Config` object cached in DataCache

**Severity:** Minor — latent risk; no current caller mutates it
**File:** [FileFinder.php:39](../../packages/kernel/core/src/Libraries/FileFinder.php#L39),
[FileFinder.php:61-63](../../packages/kernel/core/src/Libraries/FileFinder.php#L61-L63)

```php
$cfg = new Config(require $configFile, true);   // mutable: true
// ...
$cfg->setNamespacePaths($namespacePaths);        // mutates to add resolved paths
$this->config = $cfg;
$this->cacheManager->data()->set(self::class, ['config'], $cfg, true);
```

The `Config` object is created mutable so `setNamespacePaths()` can be called once during
setup. Afterwards it's stored in `$this->config` and the DataCache. Since DataCache stores
by reference (local PHP process), any code that obtains the `Config` and calls a setter on it
would silently corrupt the cached state for the entire request — and, if APCu is involved,
potentially across requests.

`discoverNamespacePaths()` is private, so external callers can't reach `$cfg` directly. The
risk is theoretical today. But the `Config` should be immutable once setup is complete.

**Fix:** Two options:
- (a) Instantiate `Config` with `mutable: false` after populating `namespacePaths` by
  creating a second immutable Config from the final data array.
- (b) Keep mutable during build, then call `$cfg->seal()` / mark immutable — not currently
  supported by `Config`, would require a small addition.

Option (a) is simpler given the current `Config` API:
```php
$cfg = new Config(array_merge(require $configFile, ['namespacePaths' => $namespacePaths]));
// mutable: false (default) — no setters possible after construction
```

### QUAL-FF007 — `if ($path)` truthiness check instead of strict null check

**Severity:** Minor — misleading; communicates wrong intent
**File:** [FileFinder.php:84](../../packages/kernel/core/src/Libraries/FileFinder.php#L84)

```php
// Current
$path = $this->cacheManager->data()->get(self::class, $key);
if ($path) {
    return $path;
}

// Fix
if ($path !== null) {
    return $path;
}
```

`DataCache::get()` returns `null` on a miss, a string path on a hit. The truthiness check
`if ($path)` would silently skip a cached empty string, which can't happen in practice
(we only cache real file paths), but the intent is "did the cache return anything?" not
"is the value truthy?". Strict `!== null` communicates the actual contract.

## Architecture Notes (no fix required, document for successors)

### A — `findFirstMatch` normalises the namespace string internally

`findFirstMatch` calls `Naming::toNamespaceString($nameSpace)` on line 79 before looking up
the cache. All current callers already pass a pre-normalised string (output of
`Naming::toNamespaceString()`), so the internal call is redundant but harmless.

This is a defensive measure: the public API accepts raw strings
(`'Z77\\Module\\Frontend\\'` or even `'z77/module/frontend'`) and normalises them. Callers
should not rely on this normalisation — pass a correctly-formed namespace. The behaviour
should be documented in the method docblock to avoid confusion.

### B — Template files use `sourcePaths`, not `assetPaths`

`getFirstTplMatch` resolves templates via `resourceType: 'sourcePaths'` with `tplDir` as
subDir. Templates are PHP files inside the `src/` tree, not in `public/assets/`. This is
correct and intentional, but the distinction is non-obvious to successors: templates are
"source files" from FileFinder's perspective, not assets.

### C — `getAbsPath` has no path traversal protection

`getAbsPath('/logs', true)` constructs `ABS_BASE_PATH . '/' . ltrim($subDir, '/')`. No
guard against `..` sequences. Currently called only with hardcoded strings in Bootstrap, so
no risk in practice. If ever exposed to user-controlled input, add a `realpath()` check and
compare the result against `ABS_BASE_PATH`.

### D — Override system is namespace-scoped, not file-scoped

The override system works at the namespace+path level, not per-file. If a project adds
`override/z77/core/src/Services/LayoutManager.php`, it shadows the entire class. There is no
partial override (method-level). Successors should understand this when planning overrides:
copy the full file, not just the changed methods.
